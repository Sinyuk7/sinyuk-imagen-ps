## **核心判断**

你现在效率低的根本原因，不是 Codex “不懂 Photoshop UXP”，而是项目缺少一个 **Agent 可以直接操控、读取和验证的调试闭环**。

Adobe 官方推荐流程仍然高度依赖人工 GUI：

`启动 Photoshop → 启动 UXP Developer Tool → Load & Watch → 打开插件 → 点击操作 → 打开 UDT Debu:contentReference[oaicite:0]{index=0}ugger 确实基于 Chrome DevTools，支持 Console、Sources、Network、断点和 Break on start；但这些信息主要存在于 GUI 内，并不会自然变成 Codex 可以检索、比较、归档的文件。:contentReference[oaicite:2]{index=2}obe 曾经提供过非常适合 Agent 的 `uxp-devtools-cli`，能执行：

```bash
uxp plugin load
uxp plugin reload
uxp plugin debug
uxp plugin logs
uxp plugin test
```

甚至支持 WebDriverIO 自动化测试。但它最后一个正式版本 `1.5.1` 发布于 2022 年 2 月，并明确不支持 Apple Silicon 原生环境，只能尝试在 Intel/Rosetta 环境运行。因此，在你的 M 系列 Mac 上，它不适合作为项目长期 Harness 的核心。([GitHub][1])-

# **一、目前工作流存在的结构性问题**

## **1. Codex 看得见代码，却看不见运行时**

Codex CLI/IDE 擅长：

* 修改文件
* 运行 `pnpm build/test`
* 读取 stdout/stderr
* 分析生成的日志和测试报告

但 UXP 的关键运行时信息在：

* UDT Console
* UDT Sources
* Photoshop 当前文档状态
* Photoshop 原生 crash report
* 插件面板的实际交互状态

这些信息没有进入项目目录，因此 Codex 每次只能猜：

> 修改一段代码 → 让你手动启动 → 点击 → 你告诉它崩了 → 再猜一次

这不是一个可收敛的 Debug Loop。

## **2. `console.log` 对进程级 crash 几乎不可靠**

普通 JavaScript 异常通常能在 UDT Debugger 中看到堆栈；但如果 Photoshop 整个进程直接崩溃：

* DevTools 可能一起关闭
* 最后几条 Console 信息可能丢失
* Promise rejection 可能还没输出
* 文件或网络缓冲区可能还没 flush
* Codex 只能拿到事后 `.ips` crash report

Adobe 的 UXP Changelog 中曾修复过：

* `fit-content` 导致插件崩溃
* 空 SVG polygon 导致 Photoshop 崩溃
* UDT Watch Reload 导致 Photoshop 频繁崩溃
* WebView 可见性脚本引起宿主 crash
* Photoshop 崩溃时 UDT Console 一同关闭

所以“Photoshop crash”不能直接等价为“你的 TypeScript 抛了异常”。UXP runtime、CSS、SWC、WebView、Reload 机制和 Photoshop native code 都可能参与。([Adobe Developer][2]) **3. 手动点击不是稳定的测试入口**

如果每次测试都必须：

1. 找到 UDT 窗口
2. Reload
3. 切换 Photoshop
4. 打开面板
5. 创建或打开特定 PSD
6. 选择某个图层
7. 点击某个按钮

那么任何一个窗口状态、文档状态、选择状态变化，都会让结果不可复现。

Agent 最需要的是：

```bash
pnpm uxp:scenario -- submit-current-selection
```

而不是：

> 请在 Photoshop 中找到这个按钮并点击。

## **4. UI、业务逻辑和 Photoshop API 调用耦合过深**

如果 React `onClick` 内部直接完成：

* 读取当前文档
* 读取选区
* `imaging.getPixels`
* Base64 转换
* Provider 请求
* `executeAsModal`
* `batchPlay`
* 写回 Photoshop
* 更新 UI 状态

那么 Codex 不启动 Photoshop，就无法验证绝大多数行为。

你目前已有的 `application / core-engine / providers / UXP app` 分层，实际上非常适合解决这个问题：**UXP 端应该进一步变薄，只做宿主适配，不承担完整业务编排。**

# **二、推荐的完整调试架构**

```text
Codex
  │
  ├─ pnpm test:headless
  ├─ pnpm uxp:bridge
  ├─ pnpm uxp:scenario -- <scenario>
  │
  ▼
Node Diagnostic Bridge
  │  localhost WebSocket
  ▼
UXP Dev Bridge
  │
  ├─ application command
  ├─ Photoshop host adapter
  └─ structured diagnostics
  │
  ▼
Photoshop
  │
  └─ JSONL / result / crash / screenshot
       写入 .artifacts/uxp/runs/<runId>/
```

核心思想是将其拆成两个平面：

* **控制面**：Codex 能触发哪个场景。
* **观测面**：Codex 能获得什么证据。

# **三、观测面：建立 UXP Flight Recorder**

## **1. 不再只依赖 Console**

在插件中增加一个统一的 `diagnostics` 模块，对所有高风险路径写结构化事件。

推荐事件字段：

```ts
interface DiagnosticEvent {
  schemaVersion: 1;
  runId: string;
  timestamp: string;
  sequence: number;

  buildVersion: string;
  gitCommit?: string;
  photoshopVersion?: string;
  uxpVersion?: string;

  component: string;
  action: string;
  phase: "start" | "checkpoint" | "success" | "error";

  durationMs?: number;

  document?: {
    id?: number;
    width?: number;
    height?: number;
    mode?: string;
    activeLayerCount?: number;
  };

  payload?: {
    byteLength?: number;
    pixelWidth?: number;
    pixelHeight?: number;
    descriptorHash?: string;
  };

  error?: {
    name?: string;
    message?: string;
    number?: number;
    stack?: string;
    cause?: unknown;
  };
}
```

重点不是记录越多越好，而是知道：

> Photoshop 崩溃前，最后成功执行到了哪一步。

例如：

```text
001 click.generate.start
002 selection.read.success
003 pixels.get.start
004 pixels.get.success
005 image.encode.start
006 image.encode.success
007 executeAsModal.start
008 batchPlay.apply.start
--- Photoshop crashed ---
```

这比一份几万行 `.ips` 日志更容易快速缩小范围。

## **2. 所有高风险调用都要写“调用前检查点”**

不要只写：

```ts
await batchPlay(...);
log("batchPlay completed");
```

因为 Photoshop 如果在 `batchPlay` 内崩溃，第二条永远不会出现。

应该是：

```ts
await diagnostics.checkpoint("batchPlay.apply.start", {
  descriptorHash,
  descriptorCount,
});

const result = await batchPlay(descriptors, options);

await diagnostics.checkpoint("batchPlay.apply.success");
```

开发模式下，即使多一次日志写入有少量开销，也比无法定位 native crash 更有价值。

## **3. 双写日志**

推荐两个 Sink：

### Sink A：本地 Node Bridge

UXP 通过 localhost WebSocket 将事件发送给 Node 进程，Node 写入：

```text
.artifacts/uxp/runs/<runId>/trace.jsonl
```

UXP 官方提供 `window.WebSocket`，但需要在 Manifest 中声明对应的 network domain。建议先做一个最小 PoC，确认你当前 Photoshop/UXP 版本允许连接 `localhost`，再正式接入。([Adobe Developer][3])# Sink B：`plugin-data://` 环形日志

WebSocket 断开或 Photoshop 直接 crash 时，将最近的 breadcrumb 同步保存到插件自己的 storage。

Manifest v5 默认允许访问：

* `plugin://`
* `plugin-temp://`
* `plugin-data://`

不需要为了诊断日志申请整个磁盘的 `fullAccess`。([Adobe Developer][4])的意义是：

* WebSocket：方便 Codex 实时读取。
* `plugin-data`：宿主崩溃后的恢复保险。

插件下一次启动时，可以把上一次未正常结束的 run 标记成：

```json
{
  "status": "host-crashed-or-terminated",
  "lastCheckpoint": "batchPlay.apply.start"
}
```

## **4. 统一包裹所有入口**

至少包裹：

* `entrypoints.setup`
* panel `create/show/hide/destroy`
* React 顶层 Error Boundary
* 所有按钮 handler
* `executeAsModal`
* `batchPlay`
* `imaging.getPixels`
* `imaging.putPixels`
* 文件读写
* Provider 请求
* Base64/ArrayBuffer 转换

不要只依赖 `window.onerror` 或 `unhandledrejection`，因为 UXP 并不等同于标准 Chromium 页面；显式 wrapper 才是最可靠的边界。

# **四、控制面：不要让 Codex 通过点击触发业务逻辑**

## **1. 将按钮行为改成 Command**

React 按钮不应拥有业务逻辑：

```ts
onClick={() => application.execute({
  type: "GENERATE_FROM_CURRENT_SELECTION",
})}
```

然后测试桥同样调用：

```ts
application.execute({
  type: "GENERATE_FROM_CURRENT_SELECTION",
})
```

这样可以保证：

* 手动点击走同一路径
* Codex 场景测试走同一路径
* CLI/mock 测试也能覆盖大部分路径

## **2. 增加开发态 Scenario Runner**

例如：

```ts
type UxpScenario =
  | "plugin.boot"
  | "panel.render"
  | "read.activeDocument"
  | "read.currentSelection"
  | "capture.activeLayer"
  | "generate.mock"
  | "apply.mockResult"
  | "full.smoke";
```

Node 发送：

```json
{
  "type": "runScenario",
  "runId": "20260623-001",
  "scenario": "apply.mockResult",
  "fixture": "1024x1024-rgb"
}
```

UXP 返回：

```json
{
  "runId": "20260623-001",
  "status": "passed",
  "durationMs": 842,
  "checkpoints": 12
}
```

于是 Codex 可以真正执行：

```bash
pnpm uxp:scenario -- apply.mockResult
cat .artifacts/uxp/runs/latest/result.json
cat .artifacts/uxp/runs/latest/trace.jsonl
```

而不是盯着 Photoshop 面板猜测。

## **3. 使用可重复 Fixture**

场景测试不要依赖你当前随便打开的一张图片。

准备少量固定 fixture：

```text
fixtures/photoshop/
├── rgb-1024.psd
├── rgb-selection.psd
├── layer-mask.psd
├── smart-object.psd
└── large-8k.psd
```

每个 Fixture 明确：

* 文档模式
* 色深
* 图层结构
* 是否有选区
* 是否有 mask
* 预计输出
* 是否属于压力测试

# **五、Codex 应该如何参与 GUI**

截至 2026 年 6 月，Codex App 的 Computer Use 已能查看屏幕、点击、输入和操作 macOS 应用，官方也明确将“重现仅出现在 GUI 中的 bug、测试桌面应用”列为适用场景。([OpenAI Developers][5])，你应该使用 **Codex App，而不是只使用 Codex CLI** 来承担以下步骤：

* 启动 Photoshop
* 启动 UXP Developer Tool
* Load 或 Reload 插件
* 打开 Debugger
* 打开插件面板
* 做最终视觉检查
* 截取 crash 前后界面

但不要让 Computer Use 承担：

* 从 Console 中复制几百行日志
* 每次手动重复业务点击流程
* 判断事件执行到哪一步
* 从 DevTools 中人工分析复杂对象
* 充当唯一测试入口

官方同样建议：当应用存在结构化 integration 时，应优先使用结构化接口，Computer Use 用于结构化能力覆盖不到的 GUI 流程。([OpenAI Developers][5])

## **推荐制作一个 Repo Skill**

目录：

```text
.agents/skills/photoshop-uxp-debug/
├── SKILL.md
├── scripts/
│   ├── collect-crash-report.sh
│   ├── start-diagnostic-bridge.ts
│   ├── run-scenario.ts
│   └── summarize-trace.ts
└── references/
    ├── failure-classification.md
    └── uxp-debug-checklist.md
```

Codex 官方支持将 Skill 放在 `$REPO_ROOT/.agents/skills`，也支持通过 Record & Replay 将一段稳定 GUI 操作录制成可复用 Skill。([OpenAI Developers][6])只录制稳定的 GUI 前置流程：

```text
启动 UDT
→ 启动 Photoshop
→ Load plugin
→ 打开 panel
→ 打开 Debugger
```

不要把具体功能测试点击也录进去。

# **六、Photoshop crash 的自动收集**

为每次 Scenario 创建独立目录：

```text
.artifacts/uxp/runs/20260623-001/
├── env.json
├── trace.jsonl
├── result.json
├── screenshot-before.png
├── screenshot-after.png
└── photoshop-crash.ips
```

Crash collector 需要完成：

1. 运行场景前记录现有 `.ips` 文件和时间。
2. 记录 Photoshop PID。
3. 运行测试。
4. 检测 Photoshop 是否异常退出。
5. 找出新生成的 Photoshop `.ips`。
6. 复制到当前 `runId` 目录。
7. 提取：

   * Exception Type
   * Termination Reason
   * Crashed Thread
   * 崩溃线程前 20 个 frame
   * Photoshop/UXP/插件版本
8. 与 `trace.jsonl` 的最后一个 checkpoint 合并。

Apple 当前通过 Console 的 Crash Reports 展示 `.ips`；对于无响应而非 crash，则应收集 `.spin`/Process Sample。([Apple Support][7])obe 也建议主要关注：

* `Crashed Thread`
* 该线程最顶部的 stack frames

而不是让 Codex逐行阅读整份 crash report。([Adobe Help Center][8])-

# **七、当前 crash 最应该检查的问题**

即使 crash 本身不是这次讨论重点，你也应先把它划分到明确边界。

## **阶段 A：Load 就崩**

重点检查：

* Manifest 与构建产物路径
* 顶层模块执行
* 全局 `require`
* React/SWC 注册
* CSS 与 SVG
* entrypoint 初始化工作量

Manifest v5 的 entrypoint lifecycle 存在 **300ms timeout**，不要在 panel `create/show` 中进行网络、像素读取或复杂初始化；先完成最小 UI 挂载，再异步启动其余逻辑。([Adobe Developer][4]) **阶段 B：打开 Panel 才崩**

重点二分：

```text
空 HTML
→ React root
→ 基础组件
→ SWC 组件
→ CSS
→ 状态初始化
→ Photoshop document subscription
```

UXP 曾发生过由 CSS/SVG/SWC runtime 引起的宿主 crash，因此 UI 层也必须纳入二分，不要只排查 `batchPlay`。([Adobe Developer][2]) **阶段 C：点击按钮才崩**

重点逐段关闭：

```text
事件 handler
→ 读取文档
→ 读取选区
→ getPixels
→ ArrayBuffer/Base64
→ Provider
→ decode
→ executeAsModal
→ putPixels/batchPlay
```

结合你的项目涉及像素数据和 Base64，尤其要记录：

* width / height
* channel count
* bit depth
* ArrayBuffer byteLength
* Base64 字符长度
* 同时存在的 buffer 数量
* 是否重复复制 buffer

这类问题可能不是普通 JS 异常，而是宿主内存压力。

## **阶段 D：进入 Photoshop 修改操作后崩**

重点检查：

* `executeAsModal` 是否完整 `await`
* 是否发生嵌套或并发 modal
* 是否吞掉了 exception
* 是否存在无中断的循环
* 用户取消后是否继续调用 `batchPlay`
* descriptor 是否包含非法、过期的 document/layer reference

Adobe 明确要求 `executeAsModal` 应使用 `await`，并说明未等待、吞掉异常、紧密循环和 modal 冲突都会导致运行状态异常。([Adobe Developer][9])-

# **八、推荐落地顺序**

## **第一阶段：先获得证据**

立即增加：

* dev source map、关闭 minify
* entrypoint/button/modal/API wrapper
* `runId + checkpoint + JSONL`
* Photoshop `.ips` 自动收集
* 固定 Fixture
* `Load` 与 `Load & Watch` 分开测试

复现 crash 时建议暂时关闭 Watch，使用显式 Reload，排除 Watch Reload 自身变量。

## **第二阶段：让 Codex 可以触发**

增加：

```bash
pnpm uxp:bridge
pnpm uxp:scenario -- panel.render
pnpm uxp:scenario -- capture.activeLayer
pnpm uxp:scenario -- apply.mockResult
pnpm uxp:collect
pnpm uxp:triage
```

Codex 的工作从“手动点按钮”变成“运行场景并读取 artifact”。

## **第三阶段：自动化 GUI 前置**

使用 Codex App Computer Use + Record & Replay：

* 启动 Photoshop/UDT
* Load 插件
* 打开 panel
* 打开 Debugger
* 确认 panel 可见

GUI 只剩少数无法结构化的步骤。

## **第四阶段：再评估 Adobe CLI**

可以做一次有明确停止条件的 Rosetta PoC：

```text
目标：验证 uxp plugin load / reload / logs 是否仍兼容当前 Photoshop
成功：可稳定运行 10 次，日志可被 Agent 读取
失败：立即放弃，不继续维护旧 CLI fork
```

不要一开始就投入大量时间修复 `uxp-devtools-cli`。它虽然功能方向正确，但年代、Node/Electron 依赖和 ARM 缺失都会变成新的维护负担。

---

# **最终建议**

你真正需要建设的不是“一个会操作 UDT 的 Codex”，而是一个小型的 **Photoshop UXP Harness**：

* 大部分逻辑可 headless 测试
* Photoshop 操作通过 Command 暴露
* 场景可由 Node/Codex 触发
* 插件运行过程产生 JSONL Trace
* Native crash 自动关联 `.ips`
* GUI 自动化只负责启动、加载和视觉确认
* 固定 Scenario 可以在修改后反复重放

一旦这层完成，Codex 的排查模式会从：

> 修改 → 猜测 → 你点击 → crash → 再猜

变成：

> 运行同一 Scenario → 读取最后 checkpoint → 定位边界 → 最小修改 → 重跑同一 Scenario → 对比结果

这才是最有价值的效率提升。
