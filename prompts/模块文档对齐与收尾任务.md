请在 <MODULE_PATH> 内执行一次模块活跃文档收敛任务。

目标：
在模块阶段性开发、OpenSpec、STATUS.md 对齐任务基本完成之后，清理当前模块的活跃文档上下文，移除历史噪音、多重真理源和过期规划文档。

本任务的核心不是创建更多文档。
本任务的核心是：对齐、收敛、删除、归档、去引用、最小化当前权威入口。

最高原则：
- 活跃模块目录只保留当前仍然有权威价值的文档。
- 历史文档不得污染 README / AGENTS / 默认阅读路径。
- 如果文档已经不是权威来源，不要只加 historical / archived 声明；必须移出活跃上下文、删除，或从所有默认入口中移除。
- 不要为了“解释历史”让未来 Agent 继续阅读历史文档。
- 如果存在真实未完成项、疑问、待确认、技术债，必须收敛到一个最小的 active ledger，而不是散落在 PRD / STATUS / SPEC 中。

## 0. Task Intent

当前模块可能残留以下开发期文档：

- STATUS.md
- PRD.md
- SPEC.md
- Design.md
- planning notes
- draft docs
- completed OpenSpec change notes
- 临时对齐报告
- 历史 checklist
- 临时风险表

这些文档在开发初期有价值，但模块稳定后可能变成：

- 过期上下文
- 重复信息
- 多重真理源
- Agent 误导源
- 默认读取噪音
- 后续维护负担

本任务要判断它们应该：

- keep：继续作为当前权威文档
- merge：少量有效内容合并进现有权威文档
- move_to_archive：移出活跃模块上下文
- delete：用户允许时直接删除
- deauthorize：只在无法移动/删除时使用，并且必须从默认入口移除
- no_action：无需处理

## 1. Operating Mode

默认模式是 audit-only：

- 只输出收敛建议
- 不移动文件
- 不删除文件
- 不修改文档
- 不修改代码或配置

如果用户明确要求 apply cleanup，才允许执行文档收敛动作。

apply cleanup 只允许：

- 删除明确无长期价值的历史文档
- 将历史文档移动到 archive
- 从 README / AGENTS / docs index 中移除历史文档引用
- 将少量长期有效内容合并进现有权威文档
- 创建或更新最小 unresolved ledger
- 更新当前权威入口中的过期文档索引

apply cleanup 禁止：

- 修改源码
- 修改 package.json / lockfile / tsconfig / build config
- 创建 OpenSpec change
- 生成 proposal / design / specs / tasks
- 重写整个 PRD / SPEC / README
- 为了完整性创建一组新文档
- 把历史文档修补到“看起来一致”

## 2. Hard Rules

必须遵守：

- 不预设必须保留 README.md / AGENTS.md / SPEC.md / TESTING.md / CONTRACTS.md 等固定矩阵。
- 不覆盖项目已有文档格式。
- 不改写 AGENTS.md 的稳定格式，除非用户明确要求。
- 不为了“文档体系完整”创建空文档。
- 不把 STATUS.md 继续维护成长期状态源。
- 不把 PRD.md / SPEC.md 的历史内容直接当成当前事实。
- 不把 archive 文档加入 README 的默认阅读路径。
- 不在 README 中写“PRD.md 是历史输入，仅供参考”作为收尾结果。
- 不在 README 中写“STATUS.md 已归档，详见 xxx”作为当前模块入口。
- 不让未来 Agent 必须阅读历史文件才能理解当前模块。
- 不做完整代码审查。
- 不修改业务逻辑代码。
- 不修改依赖配置。
- 不创建 openspec/changes/*。
- 不生成 proposal.md / design.md / specs / tasks.md。
- 不继续拆分 implementation spec。
- 不把外部模块纳入当前模块 scope。

## 3. Active Context Hygiene Rules

本任务的最高优先级是减少未来 Agent 默认上下文污染。

如果某个文档已经不是当前权威来源，不要仅仅在文件顶部加 archived / historical 声明。
不要仅仅在 README 中标注“此文件不再权威”。
不要让非权威文档继续留在模块根目录作为默认可见入口。

非权威文档必须执行以下动作之一：

1. delete
2. move_to_repo_archive
3. move_to_module_archive
4. remove_from_active_references

优先级：

- 如果用户允许删除，直接删除无长期价值的开发期文档。
- 如果仍有历史价值，优先移动到 <REPO_ROOT>/.archive/modules/<module-name>/。
- 只有项目已有模块内 archive 约定时，才移动到 <MODULE_PATH>/docs/archive/。
- 不要在 README.md 的当前文档入口中引用 archive 文档。
- 不要让 AGENTS.md 的默认读取路径包含 archive 文档。
- archive 文档只能通过 archive index、Git history 或人工考古查找，不作为日常上下文。

## 4. Context Loading Rules

先读取当前模块已有文档入口。

优先读取：

- <MODULE_PATH>/README.md
- <MODULE_PATH>/AGENTS.md
- <MODULE_PATH>/STATUS.md
- <MODULE_PATH>/PRD.md
- <MODULE_PATH>/SPEC.md
- <MODULE_PATH>/Design.md
- <MODULE_PATH>/docs/ 的 1~2 层目录索引
- <MODULE_PATH>/openspec/changes/ 的目录索引

读取策略：

- 先识别项目已有文档约定。
- 如果 AGENTS.md 已经规定文档格式或目录规则，必须优先遵守。
- 如果 README.md 已经定义模块文档入口，必须检查它是否仍然只指向当前权威文档。
- 如果 STATUS.md 已完成使命，优先判断 delete / move_to_archive / remove references。
- 如果 PRD.md / SPEC.md 已经与当前现实冲突，不要继续把它们修补成权威文档。
- 如果存在 archive 目录，默认不深入读取，除非需要确认迁移或删除对象。

不要做：

- 不递归展开所有历史 OpenSpec change。
- 不完整阅读所有源码。
- 不为了整理文档而重新理解整个系统。
- 不把旧文档的所有内容重新总结一遍。

## 5. Targeted Reality Check

只做最小代码现实检查。

允许读取：

- package.json，只读，不修改
- public exports
- index.ts / index.py / main entry
- types / interfaces / contracts / schemas
- 文档明确引用的代码路径
- 与 public contract 直接相关的测试入口

禁止读取：

- 不递归扫描 src
- 不分析复杂业务逻辑
- 不重构代码
- 不检查每个函数实现
- 不把内部实现细节写入长期文档
- 不修改 package.json 或依赖

读取代码的目的只能是：

- 确认当前文档是否过期
- 确认 public API / public contract 是否存在
- 确认模块边界是否与文档一致
- 确认某些开发期计划是否已经完成
- 确认历史文档是否还有保留价值
- 确认 unresolved ledger 是否需要记录真实未完成项

如果发现代码或配置 drift：

- 只能记录为 follow-up
- 不能在本任务中直接修改
- 不能顺手修 package.json、lockfile、源码或测试

## 6. Historical Document Handling Rules

对于 STATUS.md / PRD.md / 阶段性 SPEC.md：

不要默认修补这些文件。
不要为了让它们“看起来一致”而更新内容。
如果它们已经完成历史使命，应该移动、删除或去除引用。

处理顺序：

1. 判断是否仍是当前权威来源。
2. 如果不是，提取其中仍有长期价值的信息。
3. 将长期价值信息合并进现有权威文档，或 unresolved ledger。
4. 将原文件 delete 或 move_to_archive。
5. 从 README / AGENTS / docs index 中移除对原文件的默认引用。

禁止：

- 在 README 里写“PRD.md 是历史输入，仅供参考”。
- 在 README 里写“STATUS.md 已归档”。
- 在模块根目录保留 PRD.md / STATUS.md 并只加顶部声明。
- 为了去权威化而制造更多解释性噪音。
- 为了保留旧文件而修复所有过时段落。

## 7. README Reference Rule

README.md 只能列出当前权威入口。

不要在 README.md 的主文档入口中列出：

- archived STATUS.md
- historical PRD.md
- outdated SPEC.md
- completed planning docs
- old OpenSpec changes
- docs/archive 下的历史材料
- 已完成的阶段性对齐报告

如果确实需要保留历史索引：

- 放到 <REPO_ROOT>/.archive/INDEX.md
- 或放到 archive 目录内部
- 不要放在当前模块 README 的默认阅读路径里

README.md 的目标是减少阅读路径，不是解释所有历史路径。

## 8. SPEC Handling Rule

不要默认保留 SPEC.md。

先判断 SPEC.md 当前角色：

- 如果 SPEC.md 是当前唯一准确的 public contract / module boundary 文档，可以保留并最小修正。
- 如果 SPEC.md 是阶段性实现规格，应提取仍有效内容后 archive 或 delete。
- 如果 SPEC.md 同时包含当前事实和历史计划，应拆分：当前事实合并进现有权威文档，历史计划移出活跃上下文。
- 如果 SPEC.md 与当前代码不一致，不要优先修补 SPEC；优先判断它是否应该继续存在于活跃上下文。

目标不是让旧 SPEC 变干净。
目标是让当前模块只剩必要权威文档。

## 9. Authority Convergence Rules

不要预设稳定期必须有哪些文件。

必须先回答：

- 当前项目已经把哪些文档当成权威入口？
- 哪些文档只是历史计划？
- 哪些文档与当前代码或当前设计冲突？
- 哪些文档内容重复？
- 哪些文档会误导后续 Agent？
- 哪些文档还有长期价值？
- 哪些内容应该合并到现有文档，而不是新建文档？
- 哪些文件应该从 README / AGENTS / docs index 中移除引用？
- 哪些未完成事项需要进入 unresolved ledger？

文档收敛优先级：

1. 先保留项目已有权威入口。
2. 再合并长期有效信息。
3. 再记录真实未完成项。
4. 再归档历史文档。
5. 再删除无价值噪音。
6. 最后才考虑新建缺失文档。

新建文档必须满足：

- 现有文档无法自然承载该信息。
- 该信息有长期维护价值。
- 后续开发者或 Agent 会反复需要。
- 不创建会导致权威入口混乱。
- 用户或项目约定允许新增。

否则不要新建。

## 10. Unresolved Ledger Rule

一次文档收敛扫描后，很可能发现：

- 未完成事项
- 历史债务
- 待确认问题
- 跨模块依赖疑问
- 风险残留
- 代码与文档不完全一致
- 测试未覆盖的关键边界
- 需要后续单独处理的 cleanup

这些内容不能继续散落在 STATUS.md / PRD.md / SPEC.md 中。

必须使用 unresolved ledger 机制收敛。

优先级：

1. 如果项目已有 issue tracker、BACKLOG、TODO、MAINTENANCE、OPEN_ITEMS 等约定，遵守现有约定。
2. 如果没有现有约定，并且确实存在未解决问题，建议创建一个最小台账：
   - <MODULE_PATH>/OPEN_ITEMS.md
   或使用项目约定名称。
3. 如果没有未解决问题，不创建 OPEN_ITEMS.md。
4. 如果只是本次扫描过程记录，不是未来需要跟进的问题，放入 archive report，不进入活跃模块上下文。

OPEN_ITEMS.md 只能记录当前仍然有效的问题，不记录历史过程。

允许进入 OPEN_ITEMS.md 的内容：

- confirmed_debt：已确认仍存在的技术债或文档债
- needs_decision：需要用户或架构决策的问题
- follow_up_cleanup：不允许在本任务中直接做，但后续应处理的清理项
- external_dependency：会影响当前模块维护的外部依赖
- verification_gap：关键能力尚未验证
- doc_gap：当前权威文档缺少必要信息

禁止进入 OPEN_ITEMS.md 的内容：

- 已完成的 Planned Changes
- 已解决的 blocker
- 旧 PRD 愿景
- 旧 SPEC 草图
- 过程性分析
- “也许以后可以做”的泛泛建议
- 不影响维护的 TODO
- 单纯历史解释

OPEN_ITEMS.md 建议最小结构：

# Open Items

> Current unresolved items only. Historical planning documents are not authoritative.

## Decisions Needed
- [ ] ...

## Confirmed Debt
- [ ] ...

## Follow-up Cleanup
- [ ] ...

## Verification Gaps
- [ ] ...

每条必须包含：

- evidence:
- impact:
- owner_or_next_action:
- source:

如果无法写清 evidence 和 impact，不要写入。

## 11. Archive Report Rule

本次收敛任务可以生成一份 archive report，但它不能进入模块活跃入口。

适用场景：

- 需要保留本次清理过程
- 需要记录为什么删除/归档某些文档
- 需要给人工回看
- 不希望污染 README / AGENTS / 默认路径

建议路径：

- <REPO_ROOT>/.archive/modules/<module-name>/closure-review-YYYY-MM-DD.md

archive report 可以记录：

- 被移动/删除的文件
- 为什么不再权威
- 提取了哪些长期内容
- 哪些内容被丢弃
- 哪些内容进入 OPEN_ITEMS.md
- 本次没有处理的原因

archive report 不应被当前模块 README 引用。

## 12. What To Extract From Old Docs

### From STATUS.md

只保留可能长期有效的：

- final module boundary
- unresolved external dependency
- long-term risk
- accepted trade-off
- real verification gap
- confirmed follow-up cleanup

删除或归档：

- Planned Changes
- Suggested Next OpenSpec Change
- temporary blockers
- 已完成 open questions
- execution order
- 阶段性判断

如果 STATUS.md 已经完成使命，默认建议 delete / move_to_archive / remove references。

### From PRD.md

只保留可能长期有效的：

- 模块存在原因
- 长期 non-goals
- 调用方视角的核心约束
- 不应被代码细节覆盖的产品边界

删除或归档：

- early vision
- milestone
- 已废弃需求
- 与当前现实冲突的目标
- 过时范围描述

### From SPEC.md / Design.md

只保留当前仍成立的：

- public contract
- stable boundary
- data model
- state model
- error model
- extension point
- architecture invariant
- important trade-off

删除或归档：

- 早期草图
- 废弃方案
- implementation plan
- 已完成任务清单
- 与当前代码冲突的描述

### From OpenSpec changes

只保留：

- 最终落地的架构决策
- breaking change
- 仍然影响维护的验收约束
- 重要取舍
- 未完成但仍有效的问题，进入 OPEN_ITEMS.md

不要保留：

- 临时任务清单
- 中间讨论
- 已完成步骤
- 临时 blocker

## 13. Missing Docs Rule

不要主动建议一整套标准文档。

只有在发现明确缺口时，才建议补文档。

允许建议补充的类型：

- contract information is missing
- integration boundary is unclear
- extension rule is unclear
- testing expectation is unclear
- architecture decision is important but undocumented
- existing docs contain conflicting authority
- active unresolved items have no place to live

建议补文档时，优先问：

- 能否合并进现有文档？
- 是否已有项目约定位置？
- 是否必须新建？
- 新建后是否会增加维护负担？
- 是否可以只补一个小节，而不是新建一个文件？
- 是否应该进入 OPEN_ITEMS.md，而不是新文档？

不要输出固定文档矩阵。
不要列出一堆将来可以实现的测试细节。
不要把“可以后续实现的测试体系”提前写成当前任务产物。

## 14. ADR Rule

ADR 只用于重要架构决策。

只有满足以下条件之一，才建议 ADR：

- 决策影响模块边界
- 决策影响跨模块协作
- 决策难以逆转
- 决策涉及重要 trade-off
- 决策会影响长期维护
- 决策解释了为什么放弃某个明显替代方案

不要把普通 SPEC 内容拆成 ADR。
不要为了整齐创建 ADR。
不要把 TODO 写成 ADR。
不要把 unresolved item 写成 ADR，除非它已经形成明确架构决策。

## 15. Code / Config Mutation Ban

本任务不允许修改任何代码或配置文件。

禁止修改：

- package.json
- package-lock.json
- pnpm-lock.yaml
- yarn.lock
- tsconfig.json
- vite.config.*
- src/**
- tests/**
- build scripts
- runtime config

如果发现代码或配置 drift，例如 unused dependency、导出与文档不一致、测试缺口：

- 只能记录在 findings 或 OPEN_ITEMS.md
- 只能建议创建独立 cleanup task
- 不能在本任务中直接修改

例外：

- 用户明确要求 apply code/config cleanup。
- 用户明确指定可以修改 package.json 或源码。
