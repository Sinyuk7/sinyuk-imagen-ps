请在 <MODULE_PATH> 内执行一次模块文档对齐与收尾任务。

目标：
在模块阶段性开发、OpenSpec、STATUS.md 对齐任务基本完成之后，清理开发期文档，消除多重真理源，让当前模块只保留最少、最准、最符合项目既有约定的文档入口。

本任务的核心不是创建更多文档。
本任务的核心是：对齐、收敛、去噪、归档或删除。

如果某个文档已经完成历史使命，应归档、去权威化，或在用户允许时删除。
如果某个内容仍有长期价值，应合并进现有权威文档，而不是默认新建文件。
如果项目已有 AGENTS.md、README、docs 规范，必须优先遵守现有规范，不要自创格式。

## 0. Task Intent

当前模块可能残留以下开发期文档：

- STATUS.md
- PRD.md
- SPEC.md
- Design.md
- planning docs
- notes
- completed OpenSpec changes
- 临时对齐报告

这些文档在开发初期有价值，但模块稳定后可能会变成：

- 过期信息
- 重复信息
- 多重真理源
- Agent 误导源
- 后续维护负担

本任务要判断它们应该：

- keep：继续作为权威文档
- merge：合并进现有权威文档
- archive：移动到 archive，仅保留历史价值
- deauthorize：保留文件但明确不再权威
- delete：用户明确允许时直接删除
- no_action：无需处理

## 1. Hard Rules

必须遵守：

- 不预设必须创建 README.md、AGENTS.md、TESTING.md、CONTRACTS.md 等文件。
- 不覆盖项目已有文档格式。
- 不重写 AGENTS.md 的稳定格式，除非用户明确要求。
- 不为了“完整”创建空文档。
- 不把开发期 STATUS.md 继续维护成长期状态源。
- 不把 PRD.md / SPEC.md 的历史内容直接当成当前事实。
- 不做完整代码审查。
- 不修改业务逻辑代码。
- 不创建 openspec/changes/*。
- 不生成 proposal.md / design.md / specs / tasks.md。
- 不继续拆分阶段性 implementation spec。
- 不把外部模块纳入当前模块 scope。

本任务只允许小范围读取代码，用来确认文档是否已经过期或是否存在 public contract 事实。

## 2. Cleanup Policy

默认策略：

- 如果用户没有明确允许删除文件，优先 archive 或 deauthorize。
- 如果用户明确说“删除就是删除”，允许删除明显过期且无长期价值的开发期文档。
- 删除前必须说明删除原因。
- 如果文档可能仍有历史解释价值，优先 archive，而不是直接删除。
- archive 后的文档不得继续作为权威入口引用。

归档路径优先使用项目已有约定。
如果项目没有约定，可以建议：

- <MODULE_PATH>/docs/archive/

但不要为了归档强制创建复杂目录。

## 3. Context Loading Rules

先读取当前模块已有文档入口。

优先读取：

- <MODULE_PATH>/README.md
- <MODULE_PATH>/AGENTS.md
- <MODULE_PATH>/STATUS.md
- <MODULE_PATH>/PRD.md
- <MODULE_PATH>/SPEC.md
- <MODULE_PATH>/Design.md
- <MODULE_PATH>/docs/
- <MODULE_PATH>/openspec/changes/ 目录索引

读取策略：

- 先识别项目已有文档约定。
- 如果 AGENTS.md 已经规定文档格式或目录规则，必须优先遵守。
- 如果 README.md 已经定义模块文档入口，必须优先检查它是否仍然准确。
- 如果 STATUS.md 已完成使命，优先判断是否 archive / delete / deauthorize。
- 如果 PRD.md / SPEC.md 已经与当前现实冲突，不要继续把它们当作权威来源。

不要做：

- 不递归展开所有历史 OpenSpec change。
- 不完整阅读所有源码。
- 不为了整理文档而重新理解整个系统。

## 4. Targeted Reality Check

只做最小代码现实检查。

允许读取：

- package.json
- public exports
- index.ts / index.py / main entry
- types / interfaces / contracts / schemas
- 文档明确引用的代码路径
- 与 public contract 直接相关的测试入口

禁止读取：

- 不递归扫描 src
- 不分析复杂业务逻辑
- 不重构代码
- 不检查每个函数实现
- 不把内部实现细节写入长期文档

读取代码的目的只能是：

- 确认当前文档是否过期
- 确认 public API / public contract 是否存在
- 确认模块边界是否与文档一致
- 确认某些开发期计划是否已经完成
- 确认文档是否还有保留价值

## 5. Authority Convergence Rules

不要预设稳定期必须有哪些文件。

必须先回答：

- 当前项目已经把哪些文档当成权威入口？
- 哪些文档只是历史计划？
- 哪些文档与当前代码或当前设计冲突？
- 哪些文档内容重复？
- 哪些文档会误导后续 Agent？
- 哪些文档还有长期价值？
- 哪些内容应该合并到现有文档，而不是新建文档？
- 哪些文件应该从 README / AGENTS / docs index 中移除引用？

文档收敛优先级：

1. 先保留项目已有权威入口
2. 再合并长期有效信息
3. 再归档历史文档
4. 再删除无价值噪音
5. 最后才考虑新建缺失文档

新建文档必须满足：

- 现有文档无法自然承载该信息
- 该信息有长期维护价值
- 后续开发者或 Agent 会反复需要
- 不创建会导致权威入口混乱
- 用户或项目约定允许新增

否则不要新建。

## 6. What To Extract From Old Docs

### From STATUS.md

只保留可能长期有效的：

- final module boundary
- unresolved external dependency
- long-term risk
- accepted trade-off
- final completion summary

删除或归档：

- Planned Changes
- Suggested Next OpenSpec Change
- temporary blockers
- 已完成 open questions
- execution order
- 阶段性判断

如果 STATUS.md 已经完成使命，默认建议 archive / delete / deauthorize。

### From PRD.md

只保留可能长期有效的：

- 模块存在原因
- 长期 non-goals
- 调用方视角的核心约束
- 不应被代码细节覆盖的产品边界

删除或归档：

- early vision
- milestone
- 已废弃需求
- 与当前现实冲突的目标
- 过时范围描述

### From SPEC.md / Design.md

只保留当前仍成立的：

- public contract
- stable boundary
- data model
- state model
- error model
- extension point
- architecture invariant
- important trade-off

删除或归档：

- 早期草图
- 废弃方案
- implementation plan
- 已完成任务清单
- 与当前代码冲突的描述

### From OpenSpec changes

只保留：

- 最终落地的架构决策
- breaking change
- 仍然影响维护的验收约束
- 重要取舍

不要保留：

- 临时任务清单
- 中间讨论
- 已完成步骤
- 临时 blocker

## 7. Missing Docs Rule

不要主动建议一整套标准文档。

只有在发现明确缺口时，才建议补文档。

允许建议补充的类型：

- contract information is missing
- integration boundary is unclear
- extension rule is unclear
- testing expectation is unclear
- architecture decision is important but undocumented
- existing docs contain conflicting authority

建议补文档时，优先问：

- 能否合并进现有文档？
- 是否已有项目约定位置？
- 是否必须新建？
- 新建后是否会增加维护负担？
- 是否可以只补一个小节，而不是新建一个文件？

不要输出固定文档矩阵。
不要列出一堆将来可以实现的测试细节。
不要把“可以后续实现的测试体系”提前写成当前任务产物。

## 8. ADR Rule

ADR 只用于重要架构决策。

只有满足以下条件之一，才建议 ADR：

- 决策影响模块边界
- 决策影响跨模块协作
- 决策难以逆转
- 决策涉及重要 trade-off
- 决策会影响长期维护
- 决策解释了为什么放弃某个明显替代方案

不要把普通 SPEC 内容拆成 ADR。
不要为了整齐创建 ADR。
不要把 TODO 写成 ADR。

## 11. Apply Safety Rules

如果用户要求直接 apply：

- 只执行低风险文档清理
- 只处理本报告中明确 recommended_action 的文件
- 不修改代码
- 不创建新的 OpenSpec change
- 不新增无必要文档
- 删除文件前确认 cleanup policy 是否允许 delete
- 归档文件必须加 not authoritative 标记
- README / AGENTS 中的入口引用必须同步更新，避免死链或旧权威入口

## 12. Final Response

完成后只输出：

1. 建议保留的最小权威文档入口
2. 建议 archive / deauthorize / delete 的文档
3. 需要迁移保留的长期内容
4. 是否存在必须修正的 authority conflict
5. 是否建议执行 apply cleanup

不要重复输出完整旧文档内容。
不要生成新的长期文档正文，除非用户明确要求。
不要输出未来测试体系或二次开发体系的详细实现计划。

- 是否尊重项目已有文档约定？

- 是否识别了真正的权威入口？
- 是否移除了多重真理源？
- 是否区分了 keep / merge / archive / deauthorize / delete？
- 是否避免把历史 PRD / SPEC 当成当前事实？
- 是否只做了最小代码现实检查？
