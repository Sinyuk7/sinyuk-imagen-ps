请在 <MODULE_PATH> 执行一次“模块活跃文档收敛快速审计”。

目标：
快速判断当前模块哪些文档仍应作为活跃权威入口，哪些开发期文档应该移出活跃上下文、归档或删除，并输出一份可直接执行的整理计划。

本任务的重点是：
- 快速、准确、少读文件
- 消除多重真理源
- 减少未来 Agent 默认上下文污染
- 不把历史 PRD / STATUS / 阶段 SPEC 继续留在活跃阅读路径
- 发现真实未完成项时，统一收敛为 Open Items，而不是散落在旧文档中

## 1. Hard Rules

必须遵守：

- 不重写历史文档让它们“看起来一致”
- 不在 README 里继续解释历史文档“不权威”
- 非权威文档不要留在模块根目录污染上下文
- 不默认创建 TESTING.md / CONTRACTS.md / ARCHITECTURE.md 等固定文档
- 不修改源码
- 不修改 package.json / lockfile / tsconfig / build config
- 不创建 openspec/changes/*
- 不生成 proposal / design / specs / tasks
- 不做完整代码审查
- 不递归扫描 src
- 不递归阅读所有历史 OpenSpec change

如果发现代码或配置 drift，只记录为 follow-up，不要直接修改。

## 2. Fast Context Loading

只读取必要上下文。

优先读取：

- <MODULE_PATH>/README.md
- <MODULE_PATH>/AGENTS.md
- <MODULE_PATH>/STATUS.md
- <MODULE_PATH>/PRD.md
- <MODULE_PATH>/SPEC.md
- <MODULE_PATH>/Design.md
- <MODULE_PATH>/docs/ 1 层目录
- <MODULE_PATH>/openspec/changes/ 目录名

代码只允许小范围验证：

- package.json，只读
- public export / index 文件
- types / interfaces / contracts / schemas
- 文档明确引用的代码路径

不要为了完整理解模块而继续扩展扫描。

## 3. Core Judgement

重点判断：

1. 当前真正的权威文档入口是什么？
2. README 是否还引用了历史文档？
3. STATUS.md 是否已经完成使命？
4. PRD.md 是否还应该留在活跃模块目录？
5. SPEC.md 是当前事实文档，还是阶段性实现规格？
6. 哪些旧文档应该 delete / move_to_archive / deauthorize？
7. 哪些内容值得迁移到当前权威文档？
8. 是否存在真实未完成项，需要进入 OPEN_ITEMS 或项目已有 backlog？
9. 是否存在 authority conflict / duplicate truth / stale plan？

## 4. Decision Rules

### STATUS.md

如果 Planned Changes 已完成，STATUS.md 不应继续作为活跃文档。

优先处理：

- delete
- 或 move_to_archive
- 或从所有默认入口移除

不要只加 archived 声明。

### PRD.md

如果 PRD.md 只是历史设计输入，不应留在活跃模块根目录。

优先处理：

- delete
- 或 move_to_archive
- 或从默认入口移除

不要在 README 中写“PRD.md 仅作历史参考”。

### SPEC.md

不要默认保留。

判断：

- 如果它是当前唯一准确的 public contract / module boundary 文档，可以保留。
- 如果它是阶段性实现规格，应提取有效内容后归档或删除。
- 如果它混合了当前事实和历史计划，应只保留当前事实，历史部分移出活跃上下文。

### README.md

README 只保留当前权威入口。

不要列出：

- archived STATUS.md
- historical PRD.md
- old OpenSpec changes
- completed planning docs
- docs/archive 历史材料

README 的作用是减少未来 Agent 阅读路径，不是解释所有历史路径。

### Open Items

如果发现真实未解决事项，建议进入：

- 项目已有 backlog / issue tracker / OPEN_ITEMS
- 或 <MODULE_PATH>/OPEN_ITEMS.md

只记录当前仍有效的问题：

- confirmed debt
- needs decision
- follow-up cleanup
- external dependency
- verification gap
- necessary doc gap

不要记录：

- 已完成 planned changes
- 已解决 blocker
- 旧 PRD 愿景
- 旧 SPEC 草图
- 过程性分析
- 泛泛建议

## 5. Output Format

请只输出以下报告，不要输出长篇分析。

# Active Docs Closure Report: <MODULE_PATH>

## 1. Summary

- current_authoritative_docs:
- docs_to_remove_from_active_context:
- needs_open_items: yes | no
- main_risk:
- recommended_strategy:

## 2. Findings

用简短 bullet 输出关键发现。

每条格式：

- [type] finding → action

type 只能使用：

- authority_conflict
- stale_doc
- duplicate_truth
- active_context_pollution
- missing_open_item
- code_config_drift
- no_action

## 3. Cleanup Plan

给出可执行顺序，最多 8 步。

每步必须是文档收敛动作，例如：

1. Remove STATUS.md from README references.
2. Move STATUS.md to <REPO_ROOT>/.archive/modules/<module-name>/STATUS.md.
3. Move PRD.md out of module root.
4. Keep SPEC.md only if it remains current public contract.
5. Create or update OPEN_ITEMS.md only if real unresolved items exist.

不要写代码实现任务。

## 4. Open Items Candidate

如果存在真实未完成项，列出：

- item:
  - type:
  - evidence:
  - impact:
  - suggested_location:
  - next_action:

如果没有，写：

- none

## 5. Ready-to-Apply Checklist

列出如果用户确认执行，下一步要做什么：

- files_to_move:
- files_to_delete:
- files_to_update:
- references_to_remove:
- open_items_to_create_or_update:
- forbidden_changes:

## 6. Final Recommendation

用 3 条以内总结：

- 当前最小权威入口是什么
- 哪些历史文档应该移出活跃上下文
- 是否建议立即 apply cleanup

## 6. Quality Bar

完成前检查：

- 是否没有继续维护 STATUS.md 作为状态源？
- 是否没有让 PRD.md / STATUS.md 继续污染 README？
- 是否没有为了历史解释制造更多上下文？
- 是否没有修改源码或配置？
- 是否没有大面积扫描代码？
- 是否只输出了简单报告和整理计划？
